# Модуль local.requests — Система заявок с жизненным циклом

## Контекст
На сайте есть личный кабинет. Нужно реализовать систему заявок на консультацию с модерацией, уведомлениями и контролем переходов статусов.

## Бизнес-описание функционала

### Для авторизованного пользователя (клиента)
- Пользователь заходит в личный кабинет и видит раздел «Мои заявки» — список всех своих заявок с текущими статусами.
- Может подать новую заявку на консультацию: указать желаемую дату встречи и при необходимости прикрепить документ (ТЗ, бриф и т.п.).
- После отправки заявка получает статус **«Новая»**.
- В списке пользователь видит историю своих обращений и текущий статус каждой.
- Когда заявка завершается — клиент получает письмо на почту с итогом и комментарием менеджера.

### Для менеджера (администратора)
- Менеджер работает через административный раздел Битрикса.
- Видит все заявки от всех пользователей, может фильтровать и сортировать.
- **Жизненный цикл заявки:**
    - **Новая** → **В работе** → **Завершена**
    - **Новая** → **В работе** → **Отклонена**
- Менеджер вручную двигает статус.
- При переводе в **«Завершена»** система автоматически отправляет клиенту письмо-уведомление.
- Вернуть заявку назад из **«Завершена»** или **«Отклонена»** **нельзя** — система блокирует такие попытки.
- Все попытки некорректного изменения статуса фиксируются в журнале событий.

### Автоматика (работает в фоне)
- Если заявка находится в статусе **«В работе»**, но желаемая дата встречи прошла **более 7 дней назад**, система автоматически переводит её в **«Отклонена»** (по расписанию, без участия менеджера).

### Дополнительно
- У каждого пользователя в профиле хранится счётчик поданных заявок — обновляется автоматически при каждой новой заявке.
- В каждое письмо-уведомление система автоматически добавляет служебную строку с датой отправки и номером заявки — для трассировки.

## Что нужно реализовать

### 1. Собственный модуль
- Создать модуль `local.requests`.
- Зарегистрировать все обработчики событий (не на хитах).
- Добавить языковые фразы для отображения названия модуля в административном разделе.

### 2. Инфоблок «Заявки»
Создать инфоблок со свойствами:
- **Статус** — тип «Список» (Новая → В работе → Завершена → Отклонена)
- **Дата желаемой встречи** — тип «Дата/Время»
- **Прикреплённый документ** — тип «Файл»
- **ID пользователя-автора** — тип «Число»

Добавить **UF-поле** к пользователю:
- `UF_REQUESTS_COUNT` (integer) — количество поданных заявок.

### 3. Обработчики событий (регистрация в модуле)

#### `OnBeforeIBlockElementUpdate`
- Проверить допустимость перехода статуса:
    - Из статусов **«Завершена»** и **«Отклонена»** нельзя перейти ни в какой другой статус.
- При нарушении:
    - Отменить обновление.
    - Записать попытку в журнал событий (`CEventLog::Add`).

#### `OnAfterIBlockElementUpdate`
- При смене статуса на **«Завершена»**:
    - Отправить почтовое уведомление через собственное почтовое событие `LOCAL_REQUEST_DONE`.
    - Программно добавить агент на ближайший запуск для архивации данной заявки.
    - Записать действие в журнал событий.

#### `OnAfterIBlockElementAdd`
- При создании новой заявки:
    - Инкрементировать счётчик `UF_REQUESTS_COUNT` у пользователя-автора через `CUser::Update`.

#### `OnBeforeMailSend`
- Для писем типа `LOCAL_REQUEST_DONE` добавить в тело письма строку с датой отправки и ID заявки.

### 4. Собственное почтовое событие и шаблон
- Зарегистрировать событие `LOCAL_REQUEST_DONE` с полями:
    - `REQUEST_ID`
    - `USER_NAME`
    - `REQUEST_DATE`
    - `MANAGER_COMMENT`
- Создать шаблон с минимальным текстовым содержимым (без вёрстки).

### 5. Агент
Реализовать функцию агента `\Local\Requests\Agent::archiveExpired()`, которая:
- Выбирает заявки со статусом **«В работе»**, у которых дата желаемой встречи просрочена более чем на **7 дней**.
- Меняет им статус на **«Отклонена»** через API.
- Соблюдает ограничения:
    - Нет обращений к глобальным объектам.
    - Не использует `die`/`exit`.
    - Возвращает имя функции.
- Настроить запуск и через хиты, и через cron.

### 6. Кастомизация компонента списка заявок
В шаблоне используется `bitrix:iblock.element.list`.

#### `parameters.php`
- Добавить собственный параметр `SHOW_USER_INFO` (`Y/N`).

#### `result_modifier.php`
- Если параметр `SHOW_USER_INFO` включён:
    - Подгрузить данные пользователя-автора для каждого элемента.
    - Закешировать эти данные отдельно.
    - Ключ кеша должен зависеть от группы текущего пользователя:
        - Менеджеры видят `email`.
        - Остальные — только `имя`.

#### `component_epilog.php`
- Вывести под списком количество заявок по каждому статусу **вне кеша компонента**.
- Вёрстка — plain text или минимальный `<ul>`, без стилей.

## Дополнительные требования
- Код должен соответствовать стандартам качества для грейда **J+**.
- Использовать современное API Битрикс (D7) где это возможно.
- Все изменения должны быть изолированы в модуле `local.requests`.
- Обеспечить корректную работу при использовании как на хитах, так и на cron.
